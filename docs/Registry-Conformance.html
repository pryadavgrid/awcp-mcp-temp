AWCP · Technical Conformance Report

The Agent Radar Registry and its Conformance to the Canonical Control-Plane Schema

Aligning the temp3 Agent Radar's durable state with the absolute observability/init-db schema and the governance model of the Agent Workforce Control Plane Magazine.

Branch awcp-registry-updated· Eight steps + follow-ups · all complete· Status: verified & committed

Abstract

This report records how the temporary Agent Radar (temp3) was migrated from ad-hoc, file-based persistence to the canonical AWCP control-plane schema defined in observability/init-db. That schema is treated as the absolute source of truth; the application is permitted only to read and write it, never to create or alter it. The work proceeded in eight steps — registry persistence, durable governance events, an operator approval gate, closed-loop onboarding hooks, a token ledger, crash and onboarding journals, least-privilege roles with partition maintenance, and a wired launcher — each validated against a live Postgres carrying the schema and each mapped to the vocabulary of the Agent Workforce Control Plane Magazine. Two follow-ups then closed the last gaps: aligning the risk tiers to the canonical low/medium/high CHECK (with a fail-secure clamp) and recording Temporal onboarding completion in ops.onboarding_runs. Throughout, the radar retains a fail-open posture: with no database reachable it falls back to its prior in-memory / JSON behaviour, so adoption never makes the control plane worse.

§Change summary — before vs after

What each step actually changed, end to end — the prior behaviour and what replaced it, including the two follow-ups.

Table 1 — Per-step impact: the state before the change and after it.
StepBeforeAfter
Registry persistence Before: Live registry in memory, snapshotted to a local agent_radar_registry.json file; visible only to that one process; no shared or queryable store of who exists. After: Mirrored to canonical registry.agents (text[]/jsonb/timestamptz mapped, user→os_user); restored from the DB on restart. JSON kept only as a fail-open fallback.

2 · Approval gate Before: A scope-expansion hold lived only in an in-memory approval_state field — lost on restart, since registry.agents has no approval column. After: Durably backed by governance.approval_tokens (pending→approved/denied, expiry, context_diff); the gate is rehydrated from open tokens on startup, so it survives a restart.
3 · Hooks & TTL Before: Hook flags could be trusted on declaration; degradation records (where present) lacked the from→to transition; the closed loop was unverified against the schema. After: Flags are earned by observation with last_*_ts persisted; the TTL reconciler re-quarantines silent hooks (→evidence.ledger); degradation rows carry from_profile/to_profile/trigger. Verified end-to-end.
4 · Token ledger Before: Per-call token/cost accounting lived only in memory plus an optional JSONL file. After: Each call is also mirrored to canonical evidence.token_ledger (append-only); the in-memory sliding-window/budget view is unchanged.
5 · Freeze & onboarding Before: The freeze journal was a local JSON file only; onboarding-run state existed only as a field on the agent row. After: Freezes dual-write to registry.freeze_journal (JSON kept as the crash-recovery source); onboarding runs recorded in ops.onboarding_runs (running→done, finished_at stamped).
6 · Least privilege & partitions Before: The app connected as the DB superuser, so the evidence append-only GRANT was not actually enforced; partitioned tables had only the single 2026_06 partition. After: The app connects as awcp_app (DML; DELETE/UPDATE on evidence and DDL denied); a separate admin URL creates the current + 2 monthly partitions at startup.
7 · Launch wiring Before: The launcher started the stack but did not bring up or wait for Postgres; a stale ad-hoc :5434 container (no schema) shadowed the real one. After: The launcher starts the canonical Postgres and waits for pg_isready before the gateway, so the registry persists from the first request; the stale :5434 container + orphaned volume were removed — one postgres on :5432.
8 · Tests Before: Only ad-hoc manual checks; no committed tests for the durable layer. After: tests/radar/ — a fail-open tier (11 tests, runs with no DB / no SQLAlchemy) and an opt-in Postgres integration tier (22 tests).

After: Default is low,medium,high; authoritative_risk clamps any out-of-set result to the most restrictive known tier (fail-secure), so a resolved tier can never violate the CHECK. A declared-critical agent resolves to high and persists cleanly.

1.Premise & governance model

This is a living conformance record for aligning the temp3 Agent Radar with the absolute AWCP control-plane schema shipped in observability/init-db — the concrete form of the governance model described in Agent-Workforce-Control-Plane-Magazine.html.

The magazine defines the durable governance surface as an Evidence Ledger — “workflow identity, actor, policy result, approval token, tool call, context snapshot hash, degradation state, replay trace, rollback pointer” — complemented by Approval Tokens for gated writes and Graceful Degradation for autonomy reduction. The canonical Postgres schema is precisely that surface made concrete. Conformance therefore reduces to a single principle: route the radar's state onto these canonical tables rather than onto ad-hoc ones.

2.The canonical schema

Source of truth: observability/init-db/02-schema.sql. Four schemas partition the control plane's durable state by concern:

Table 2 — Canonical schemas and their tables.
SchemaTablesConcern
registryagents, freeze_journal, gateway_agentsWho exists, who is frozen, who is gatewayed
governanceapproval_tokens, policy_decisions (partitioned), degradation_events (partitioned)Approvals, gate decisions, autonomy reduction
evidencetoken_ledger (partitioned), ledger (partitioned, hash-chained)Append-only audit & token/cost trail
opsonboarding_runs, artifactsOnboarding workflow state, stored artifacts
Ownership invariant The schema is owned by init-db — psql scripts that run once on a fresh volume — not by Alembic and not by the application. Application code only reads and writes; it never issues CREATE or ALTER. The one exception, monthly partition creation, is performed through a separate admin connection (see §9).

3.Registry persistence Done

src/awcp/radar/store.py mirrors the live registry to registry.agents instead of the local JSON file. The mapping between the Pydantic model and the SQL columns is handled explicitly:

  • AgentEntry.user → the os_user column;
  • epoch float timestamps ↔ timestamptz (via to_timestamp() on write, EXTRACT(EPOCH …) on read);
  • Python lists and dicts ↔ text[] and jsonb.

The JSON file remains a fail-open fallback whenever no database is reachable, and — matching the prior contract — only source='self' entries survive a restart; scanned processes are re-detected live. A full-sync on persist upserts the in-memory set and prunes rows no longer present.

4.Step 1 — Durable governance events Done

src/awcp/radar/db.py no longer uses a flat governance_events table; that table, alembic.ini, and migrations/ were all removed. The audit-worthy subset of events is now routed, by kind, to the canonical tables:

Table 3 — Event routing, expressed in the magazine's vocabulary.
Magazine conceptEvent kindsCanonical table
Gate Write Actions (denial)gate_deniedgovernance.policy_decisions
Graceful Degradationdegraded, autonomygovernance.degradation_events
Evidence Ledgerapproved, scope_added, scope_removed, registered, announced, removed, hook_stale, telemetry_observed, policy_observed, flags_observed, token_hard_stop, token_remote_stop, token_recover, gateway_bypassevidence.ledger
  • Data-driven routing. A _ROUTE dictionary plus a default table; adding a route is one entry, not a new branch. Nothing about the connection (host, port, credentials) or table names is hardcoded outside db.py — the URL comes from AGENT_RADAR_DATABASE_URL.
  • Hash-chained evidence. Each evidence.ledger row stores prev_hash (the previous row's hash) and row_hash (sha256(prev_hash + canonical row body)), giving a tamper-evident, append-only trail exactly as the magazine prescribes.
  • NOT NULL columns honoured. evidence.ledger.workflow_id falls back to the agent id when an event carries no workflow; policy_decisions.tool_call and decision are always supplied. The numeric columns policy_decisions.risk and degradation_events.trace_sampling are left NULL — the risk tier string lives in the jsonb payload, not the numeric column.
  • Full context preserved. Every row's payload jsonb carries the complete event; typed columns are populated best-effort (e.g. to_profile parsed from the degraded detail). The original kind is stashed under payload._kind so the unified audit read can recover it.
  • Unchanged public API. GET /events/audit reads a UNION ALL across the three tables and returns the same {ts, agent_id, event_type, payload} shape as before.
  • Fail-open. If the URL is unset or the schema is unreachable, durable logging switches off and the radar runs on its in-memory ring (GET /events).

5.Step 2 — Approval gate Done

The operator gate for scope expansion is now durably backed by governance.approval_tokens — the magazine's “New write scopes … Human approval.”

  • Scope add. A /agents/register call carrying new write_scopes creates a pending token: action_class='scope_expansion', the new write_scopes, context_diff = {added, previous, new}, and expires_at = now() + AGENT_RADAR_APPROVAL_TTL (default 7 days, env-configurable). It is inserted after the agent is persisted so the agent_id foreign key to registry.agents holds.
  • Approve. POST /agents/{id}/approve sets the open token(s) to status='approved' with decided_by / decided_at, so the gate does not re-arm.
  • Restart rehydrate. Because registry.agents has no approval column, the radar re-arms the in-memory gate on startup from any unexpired pending tokens (open_approval_agent_ids()), restoring the cross-restart durability that was lost when the registry moved off JSON.
temp3 tailoring AgentEntry.approval_state is kept as the live runtime gate so the gate also works with no database; the token table is its durable backing and restart source of truth. Canonical status CHECK values are honoured (approved / denied); the numeric risk column is left NULL.

6.Step 3 — Closed-loop hooks & TTL reconciler Done

With last_*_ts now persisted in registry.agents, the closed loop was verified end-to-end and autonomy degradation was completed into governance.degradation_events.

Gap 3Observation flips — the bit is earned, not declared

  • policy ← a /agents/{id}/gate call (_observe_policy);
  • telemetry ← an execution signal / tasks/execution event (_observe_telemetry);
  • flags ← an execution event carrying flag state (_observe_flags).

Defaults AGENT_RADAR_REQUIRE_OBSERVED_TELEMETRY/POLICY=true (flags false) are stricter than the README's all-off dev default, but match the magazine's onboarding gate — declared and observed in execution. Verified: a fresh agent stays quarantined until both a gate call and a signal are seen, then becomes active, with last_policy_ts / last_telemetry_ts persisted.

Gap 4TTL reconciler

_telemetry_reconciler drops a proven hook that goes silent past AGENT_RADAR_TELEMETRY_TTL and re-quarantines the agent, emitting hook_stale → evidence.ledger.

Conformance fixDegradation transitions

Autonomy-ladder downgrades now record full transitions. The three emit sites pass from_profile / to_profile / trigger (token_budget, failure_budget, operator) so rows are complete per the magazine's from→to model. Verified: active → trace_boost, trigger=failure_budget.

Deliberate simplifications A single AGENT_RADAR_TELEMETRY_TTL governs all three hooks (the README split it into per-hook TTLs); one TTL is sufficient and simpler. And hook_stale — a re-quarantine, i.e. a trust/status change — routes to evidence.ledger, not degradation_events, which is reserved for autonomy-profile downgrades.

7.Step 4 — Token ledger Done

The laminar TokenLedger keeps its live sliding-window and lifetime view in memory — the monitoring half, unchanged — while each call is now also mirrored to evidence.token_ledger, the magazine's Evidence Ledger token trail.

  • TokenLedger.record() calls a new _append_db() alongside the existing JSONL seam → db.record_token_usage() inserts {ts, agent_id, task_id, step, model, input_tokens, output_tokens, cost} with the real call timestamp.
  • A lazy import of awcp.radar.db from awcp.laminar.ledger keeps laminar decoupled from the radar DB layer — no import cycle.
  • evidence is append-only by GRANT, so this is INSERT-only. Fail-open: a no-op when no DB is reachable; the in-memory view and budgets are unaffected.

8.Step 5 — Freeze & onboarding journals Done

  • registry.freeze_journal. _journal_set / _journal_clear now dual-write to the canonical table (record_freeze / clear_freeze, kind ∈ process|remote). The local JSON journal stays the crash-recovery source — readable even when the DB is down, which is the more robust choice for restart repair; the table is the queryable mirror.
  • ops.onboarding_runs. record_onboarding_run upserts run state keyed by workflow_id: a Temporal start records running (real workflow id); inline completion records done and stamps finished_at, keyed by the real id or a stable inline-<agent> key. state ∈ pending|running|done.
Known gap (documented) The Temporal onboarding worker marks its run done elsewhere; this file records the running start, so a Temporal run shows running until that worker path is wired. The common inline path is fully recorded.

9.Step 6 — Least privilege & partition maintenance Done

  • Least privilege. The application connects as awcp_app — the DML role from init-db/01-roles.sql — not the superuser. This makes the canonical guarantees real: it was verified that awcp_app is denied DELETE/UPDATE on evidence.* (append-only) and denied DDL. A separate AGENT_RADAR_DB_ADMIN_URL (the owner) is used only for partition DDL at startup. Both URLs default to the init-db credentials and are fully env-overridable.
  • Partition maintenance. ensure_partitions() runs at init() through the admin connection and creates the current plus AGENT_RADAR_PARTITION_MONTHS_AHEAD (default 2) monthly partitions IF NOT EXISTS for all four partitioned tables, so inserts never hit a missing partition across a month boundary. Dates and identifiers are computed from a fixed internal list, never user input. Best-effort: a warning, not a crash, if the role cannot issue DDL.

Verified: the app runs fully as awcp_app; partitions 2026_06 / 07 / 08 were auto-created.

10.Step 7 — Launch wiring Done

  • run_everything.sh now starts the canonical postgres service with the rest of the stack and waits for it to be ready (pg_isready, up to 30 s) before the gateway — so the registry persists to registry.agents from the first request rather than falling back to JSON. Non-fatal: if Postgres never comes up, the radar still runs fail-open.
  • The stale ad-hoc awcp-postgres container (:5434, no schema) and its orphaned volume were removed; the compose now owns a single postgres on :5432 with the init-db schema.

Verified end-to-end against the real compose instance: the schema auto-applied (all four schemas), the app connected as awcp_app, the registry persisted, and partitions were present. The test volume was wiped so the first real launch gets a clean init-db apply.

11.Step 8 — Tests Done

tests/radar/ covers the durable conformance layer in two tiers:

  • Pure / fail-open (test_db_conformance.py, test_store_persistence.py) — run with no database and even without SQLAlchemy installed: the routing map, partition month-range maths (including year rollover), the durable-event whitelist, every write as a no-op when no DB, enum guards, and the registry JSON fallback (self survives a restart, scan does not, remove persists). 11 pass / 11 skip with no DB.
  • Integration — skipped unless AGENT_RADAR_TEST_DATABASE_URL points at a Postgres with the init-db schema: event routing into the three canonical tables, decision='denied', the evidence hash chain, audit filters, approval token request→decide, evidence.token_ledger, freeze upsert/clear, onboarding running→done, and partition creation. 22 pass against a throwaway Postgres.

12.Follow-up items Done

  1. Risk-tier alignment. The Gap-1 RISK_ORDER default is now low,medium,high — matching the registry.agents.risk CHECK and the magazine. authoritative_risk additionally clamps any out-of-set result (e.g. a self-declared critical with no magazine opinion) to the most restrictive known tier — fail-secure — so a resolved tier can never violate the canonical CHECK. Verified: a declared-critical agent resolves to high and persists cleanly.
  2. Temporal onboarding completion. The admit activity now records its run done in ops.onboarding_runs (keyed by the manager-assigned workflow_id, with finished_at stamped), closing the gap where a Temporal run previously stayed running. Verified end-to-end.

No conformance items remain open.

Design throughline. Every step preserves three properties: the canonical schema is never modified by the app; durable state is mirrored, never replaced, so the live in-memory views and budgets keep working; and the whole layer is fail-open, so a missing or unreachable database degrades the radar to its prior behaviour rather than breaking it.
