1Register Agents & Attach Control Hooks

Before an agent is allowed to take write actions, the control plane needs to confirm three things: that the agent's telemetry is actually running (not just claimed), that it actually calls the policy gate before acting (not just says it will), and that it actually reads feature flags at runtime (not just declares them). Step 1 is the onboarding process that checks all three by watching what the agent does in practice. An agent that hasn't proved all three stays in quarantine — it shows up in the dashboard and can read things, but cannot make any changes until it passes.

01

Registry · Onboarding · Quarantine

Agent Registry · OTel · Feature Flags
Magazine asks
Register each agent with owner, runtime, declared write scopes, feature flags, telemetry, and policy callbacks. Under-instrumented agents stay visible but cannot execute governed write actions until onboarding is complete.
Built in temp2
  • Agent registry — in-memory + JSON persistence, owner/runtime/version/scopes/flags.
  • Auto-detection — framework-agnostic scanner (module pattern + LLM-port behaviour + self-register).
  • Onboarding pipeline — map_identity → quarantine_check → link_mcp → admit (Temporal + inline fallback).
  • Quarantine gate — decide_status requires telemetry + feature flags + policy callbacks; quarantined agents are write-blocked, read-allowed.
  • Observed-telemetry loop — telemetry is now proven by observation, not just declared; auto-promote on observed events, auto re-quarantine on silence (TTL).
  • OTel hooks — setup_otel wired across every service.
  • Declared-write-scope enforcement — the gate now checks the action's scope against the agent's declared write_scopes; a write outside a granted scope is denied (out_of_scope) regardless of autonomy. Backward-compatible (no scope supplied ⇒ no check).
  • Policy callbacks observed in execution — per the magazine ("policy callbacks … observed in execution"), a declared callback list is no longer trusted: the agent must actually consult policy (call the gate) to prove the hook (policy_observed). Env-toggle AGENT_RADAR_REQUIRE_OBSERVED_POLICY restores trust-on-declare.
  • Feature-flag wiring observed in execution — the third hook now closes the same loop: an execution event that carries flag state (extra.feature_flags) proves flags_observed; otherwise the declared flags suffice. Defaults to trust-on-declare (so agents that don't yet report flags aren't penalised) and is enforced with AGENT_RADAR_REQUIRE_OBSERVED_FLAGS=true. All three onboarding hooks — telemetry, policy callbacks, flag wiring — are now "observed in execution."
Left to build
  • Nothing — Step 01 is complete. Every magazine registration field is captured, under-instrumented agents are quarantined & write-blocked-but-visible, writes are confined to declared scopes, and all three control hooks (telemetry · policy callbacks · flag wiring) are proven by observation in execution, each independently configurable. Nothing hardcoded — keyed on agent_id + env flags.
radar/store.py · radar/models.py (telemetry/policy/flags _observed + ts) · radar/api.py (GateRequest.scope, _observe_telemetry/_observe_policy/_observe_flags, hook reconciler, register init) · radar/policy.py (scope check) · radar/onboarding.py (decide_status) · radar/scanner.py · radar/detectors/*

2Orchestrate Existing Workflows

Step 2 asks whether the control plane can adopt an agent that is already running — not one it started, but one that was already mid-run in some framework like LangGraph or CrewAI when the control plane came up. The magazine's vision is to pick up that agent without restarting it: read where it is in its workflow, understand its current state, and start governing from that point forward. temp2 can fully orchestrate agents it launches from scratch (each step is tracked as a Temporal activity), but it cannot yet adopt an agent that was already in motion before the control plane started.

02

Temporal orchestration · Runtime adapters

Temporal · Runtime Adapters · Workflow DAGs
Magazine asks
Ingest active workflows from existing runtimes, normalize state, and preserve ownership and checkpoints across branches. The control plane does not replace the underlying execution stack.
Built in temp2
  • Durable orchestration — Temporal onboarding + execution workflows, in-process workers, two namespaced task queues.
  • Step-by-step driving — each agent step (reason/tool/generate) surfaced as its own Temporal activity, dynamically (no hardcoded sequence).
  • Runtime adapter — MCP gateway drives agents over stdio/SSE; the control plane decides when a tool runs.
Left to build
  • Ingest & normalize EXISTING external workflows — temp2 drives agents it orchestrates and observes scanned ones, but does not adopt/normalize in-flight state from foreign runtimes.
  • Checkpoints across branches — no branch-level checkpoint/ownership preservation beyond Temporal's own history.
  • Workflow DAGs — execution is a dynamic event loop, not a declared DAG with branch semantics.
radar/temporal/workflows/{onboarding,execution}.py · radar/temporal/activities/* · temporal/workflows/agent_execution.py · mcp/server.py · temporal/activities/mcp_gateway.py

In plain terms: if a LangGraph agent has been running for ten minutes before the radar started, the radar can see it in the registry (via scanning) but cannot take over its execution, read which step it is on, or resume it from a safe point. The work to close this gap is normalising the foreign framework's state into Temporal's model without restarting the agent.

3Gate Write Actions

The gate already stops quarantined and degraded agents from making writes. What the magazine adds on top of that is a narrower, time-limited permission system for high-risk actions. Even a fully trusted, fully onboarded agent should not be able to make a high-risk write without also holding a specific short-lived token issued for that exact action, at that exact point in the workflow, that expires after a few minutes and can only be used once. This prevents an agent from accumulating permissions, replaying an old approval for a different action, or using a legitimate token for something broader than it was issued for. OPA (Open Policy Agent) is the magazine's proposed engine for writing those policy rules as code, so they can be changed without redeploying the radar.

03

Write-action gate · Approval tokens

OPA · Approval Tokens · Tool Policies
Magazine asks
Evaluate every state-changing tool call against policy. High-risk actions pause until a narrow, expiring approval token is issued for the relevant workflow step and branch.
Built in temp2
  • Write-action gate — policy.evaluate_action: reads allowed, writes denied for quarantined / degraded / out-of-position agents.
  • Position-based ladder gating — deny logic keyed on the agent's autonomy rung, by risk, nothing hardcoded.
Left to build
  • OPA / policy-as-code — the gate is imperative; no Rego policy engine. (A separate awcp-opa prototype exists but is NOT integrated.)
  • Expiring approval tokens — the magazine's signature mechanism for high-risk writes is absent; no human-approval token flow.
  • Per-tool policies — no fine-grained per-tool / per-scope policy rules.
  • Human-approval surface — no operator approval issuance for new write scopes / cross-system writes.
radar/policy.py (evaluate_action) · radar/api.py (/agents/{id}/gate)  |  gap: OPA + approval-token issuer/validator

In plain terms: today a gate response of "allowed" means the agent is free to proceed with any write it has declared in its scope. The approval-token layer adds a second check on top: even an agent in good standing must present a narrow, expiring pass for that specific write. An approval token is just a short-lived signed string — the gate issues it, the agent sends it with the write request, and the gate validates it hasn't expired or been used before. OPA would let operators edit those policy rules in a text file without touching the Python code.

4Degrade Autonomy Gracefully

When an agent starts showing problems — too many tool failures, token budget overruns, unexpected outputs — the worst response is to flip between "fully trusted" and "completely stopped." The magazine describes a six-rung staircase that steps down gradually. First you just increase the logging detail so you have more evidence of what went wrong. Then you tighten how many things the agent can attempt in parallel. Then you switch it to a safer, more conservative model profile. Then you move it to suggestion-only mode where it can propose actions but not execute them. Only as a last resort do you suspend it entirely. Every rung is reversible if the agent recovers. temp2 has built and wired this complete six-rung ladder.

04

Degradation ladder · Token & failure budgets

Trace Sampling · Feature Flags · Safe Profiles
Magazine asks
At first failure signals: increase trace sampling, tighten retry/concurrency, shift to safer profiles, then recommendation-only before a hard stop. Each workflow can override thresholds and ladders by risk.
Built in temp2 ladder now FULL
  • Full magazine ladder — active → trace_boost → throttled → safe_profile → recommendation_only → suspended: the graceful, still-write-capable rungs (raise trace sampling → tighten retry/concurrency → shift to a safer profile) now exist before writes are blocked, exactly as the magazine sequences them.
  • Per-stage directives — each rung carries trace_sampling / max_retries / max_concurrency / profile / writes (STAGE_SPECS).
  • Live directive delivery — the runtime receives its current stage directives on every channel it polls: the gate response, the /signal response (at the exact moment autonomy drops), the /tasks/execution/*/event response, and GET /agents (effective_stage) — so it applies tighten-retry / lower-concurrency / safer-profile / raise-sampling for its next step.
  • Increase-sampling effect applied by the control plane — a degraded agent's steps are captured as evidence events (degraded_step), the magazine's "increase trace sampling" delivered as more capture when things go wrong.
  • Write-capable graceful degradation enforced — the gate allows writes on the early rungs and blocks only from recommendation_only onward; last rung = hard stop. Fully backward-compatible (legacy 3-rung / custom ladders unchanged).
  • Failure & token budgets → ladder — apply_signal / laminar breach now step through all six rungs gracefully, by risk, recoverable.
  • Immediate HARD STOP layer — process SIGSTOP, remote suspend webhook, and a token-aware LLM gateway that 429s an over-budget agent's model call.
  • Crash-recovery journal — a SIGKILLed radar no longer orphans a frozen process.
Left to build
  • Ladder is complete. The control-plane half — define the stages, enforce write-capability + hard stop, deliver the directives live, and apply the increase-capture effect — is fully done.
  • Agent honouring the delivered directives is cooperative — the agent runtime must apply the retry/concurrency/sampling it's handed, exactly as it must honour the gate. That's inherent to a control plane that doesn't run the agent's loop (magazine: runtime is "replaceable"), not a missing control-plane feature.
  • Source-PID identity is local + privilege-bound (plumbing for the hard-stop extension, not a magazine ask).
radar/policy.py (STAGE_SPECS, stage_effects, _write_block_index, evaluate_action, effective_stage, apply_signal) · radar/api.py (effective_stage on gate / signal / event / agents + degraded_step capture) · laminar/{bridge,budget,ledger}.py · radar/llm_gateway.py · laminar/estimator.py · laminar/budget.py (project()) · laminar/bridge.py (pre_check()) ★ new

In plain terms: the control plane defines the six rungs, enforces write-blocks at the right point, and delivers the current rung's instructions (lower your retry count, switch to this model, stop initiating writes) on every response channel the agent polls. The agent's job is to apply those instructions at its next step. That cooperative part is intentionally the agent's responsibility — the magazine describes the runtime as replaceable, and the control plane's job is to issue the directives clearly, not to reach inside the agent's loop.

5Replay & Recover

If an agent run fails halfway through, two questions need clear answers: exactly what did this agent do, in what order, and was each step authorised? And is it safe to resume from step four without risking doing something twice? The magazine's answer is a durable Evidence Ledger — a single joined-up record that stores the agent's identity, each tool call it made, the policy decision that allowed it, the approval token used, and a hash of the context at each checkpoint. The hash proves nothing changed between then and now, making it safe to replay. temp2 has fragments of this (Temporal stores full workflow history, the radar keeps a ring of recent governance events) but not the unified ledger with hashes and rollback pointers.

05

Evidence ledger · Checkpoints · Replay

Evidence Ledger · Checkpoints · Replay
Magazine asks
A replayable ledger with step history, context hashes, degradation state, and safe resume points, so failed branches recover without restarting the full workflow. Evidence = workflow identity, actor, policy result, approval token, tool call, context-snapshot hash, degradation state, replay trace, rollback pointer.
Built in temp2
  • Temporal history — full, replayable workflow/activity event history.
  • Recent-decisions log — in-memory ring of governance events (onboard/gate/degrade/token/telemetry).
  • Laminar JSONL trail — optional per-call token evidence (when LMNR_LEDGER_PATH set).
  • Crash-recovery journal — durable record of applied stops, replayed on restart.
Left to build
  • Unified Evidence Ledger — no single durable ledger tying identity + actor + policy result + approval token + tool call + context-hash + degradation state + replay trace + rollback pointer. Today these live in fragments.
  • Context-snapshot hashes — not captured.
  • Checkpoints / safe resume points — no branch-level checkpoint or rollback-pointer system.
  • Evidence packets / export — no operator-facing evidence bundle.
Temporal UI/history · radar/api.py (/events ring) · laminar/ledger.py (JSONL)  |  gap: durable unified Evidence Ledger + checkpoints

In plain terms: if an audit today asks "show me every action agent X took last Thursday, prove what the policy said at each step, and confirm it is safe to re-run from step three" — you would have to manually cross-reference Temporal's workflow history UI, the in-memory events ring on the radar, and optionally some JSONL log files. The Evidence Ledger puts all of that in one place with a cryptographic proof (the context hash) that the state at each checkpoint is exactly what it was when the agent ran it.

6Generate Instrumentation Patches

Any agent that hasn't set up its telemetry, policy gate calls, and feature flags will stay in quarantine indefinitely — unless a developer manually goes into that agent's codebase and adds those missing hooks. Step 6 makes the control plane do that work automatically: it detects exactly what is missing, generates a code change with the required instrumentation already written, and proposes it as a pull request. Once the developer reviews and merges that change, the agent's quarantine is lifted. In temp2, the detection is correct — the registry already identifies precisely which hooks each quarantined agent is missing — but nothing yet writes or proposes the code fix.

Left to build

These are not steps in a sequence — they are permanent infrastructure. Think of the six steps as the workflow the platform runs, and these as the structural components the workflow depends on. If one of these is missing or partial, every step that relies on it is limited.

+Built beyond the magazine (extensions)

The magazine assumes agents cooperate: they check the gate, they respect denials, and they stay within their token budget. Building temp2 with real agents quickly exposed a gap — some agents don't cooperate. They call models directly, ignore 429 responses, or proceed with writes regardless of what the gate returns. The features below were added to handle those cases, using harder stops that work regardless of whether the agent respects the gate or not. The original four were built before this session; the four marked ★ new were added this session.

Forceful token-limit enforcement

These exist in temp2 but are not described in the magazine. They were added because real autonomous agents ignored the cooperative gate. The magazine assumes onboarded agents that pass through the gate; these handle the ones that don't.

  • Process SIGSTOP / SIGCONT — freeze an over-budget local agent's OS process (pid recovered from its registry id), reversible on recovery.
  • Remote suspend/resume webhook — the network analog of SIGSTOP for agents with a control endpoint.
  • Token-aware LLM gateway — proxies model calls and 429s an over-budget agent at the source (the one wall that works for any topology). Extended this session with a pre-execution estimate (see below).
  • Crash-recovery journal — on-disk record so a killed radar never orphans a frozen process.
  • Pre-execution budget pre-check ★ new — provider-agnostic TokenEstimator hierarchy (TiktokenEstimator via tiktoken BPE → FallbackEstimator chars÷4) estimates input tokens from the raw request body before the upstream model call. bridge.pre_check() projects window_usage + estimated against the agent's per-risk-tier budget and returns HTTP 429 if the projection would exhaust it. Fail-open: any control-plane or estimation error lets the call through. Toggle: AWCP_GATEWAY_PRECHECK=false. Model→encoding routing: gpt-4o/o1/o3/o4 → o200k_base · gpt-4/claude- → cl100k_base · all others → gpt2.
  • Self-hosted Laminar observability platform ★ new — full lmnr.ai stack (Postgres · ClickHouse · Quickwit · query-engine · app-server · Next.js frontend) added to observability/docker-compose.yml alongside the existing Grafana/Tempo/Loki services on the shared awcp-obs network; host ports remapped to avoid conflicts. OTLP gRPC ingestion at :8881; web UI at :5667. Span export from awcp.laminar is now fully end-to-end to the local platform.
  • One-command stack runner ★ new — bash scripts/run_everything.sh (the single launcher) brings the whole platform up behind one port (the gateway, :8000) in dependency order: venv → Docker telemetry stack + Postgres → Temporal → Ollama → MCP → React UI → gateway (which runs the onboarding/execution workers in-process), with graceful Ctrl+C shutdown of everything it started.
  • Radar Laminar key panel ★ new — secure in-browser API-key entry in the Agent Radar UI (:8090). Key stored in sessionStorage only, never sent to the server, never shown in plain text after entry. Hides the input after saving and shows an "Open Laminar UI →" button; "change key" restores the form.
What is still missing — in plain terms. The three biggest gaps follow a natural build order. First, the approval-token gate (Step 3) — this is self-contained and can be built without waiting for anything else. An approval token is a short-lived signed string: the gate issues it, the agent sends it with the write, the gate validates it has not expired and has not already been used. OPA makes the rules behind that decision editable as text files. Second, the Evidence Ledger (Step 5) — most of the raw data already exists across Temporal history, the events ring, and optional JSONL files; the work is joining those sources into one queryable record and recording a context hash at each step so replay is safe. Third, instrumentation-patch generation (Step 6) — the hardest to automate, and the one that benefits most from the ledger already existing, since the ledger tells you exactly which hooks fired and which didn't. The teal extensions (gateway, pre-execution check, SIGSTOP, Laminar platform) are all working and do not block any of these three gaps from being closed.
Reading the scorecard. The governance spine the magazine cares about — register/quarantine (1), durable orchestration (2), graceful degradation (4) — is largely done. The biggest gaps are the magazine's two most distinctive capabilities: expiring approval tokens + OPA (Step 3) and a durable Evidence Ledger + checkpoints (Step 5), plus instrumentation-patch generation (Step 6). Most recent effort went into a forceful token-enforcement layer that is genuinely useful but sits beyond the magazine's prescribed, cooperative, policy-based model.
