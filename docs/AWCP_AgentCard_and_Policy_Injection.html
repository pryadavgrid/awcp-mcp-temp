Overview & how they relate

Two features, each adding a layer to the agent registry, deliberately kept independent:

FeatureQuestion it answersWho writes it
JSON Policy injectionWhat is each agent / tool allowed to do, and at what risk?The operator (a human), in the Policy tab.
AgentCardWhat is each agent, and what skills does it have?The agent (self-published manifest).

They meet at one point: the policy can match agents by a skill the card declares (covered in section C). The card is description; the policy is the decision. The card never decides; the operator policy is the only thing that overrides governance.

A. JSON Policy injection — what it is

An operator policy is a single JSON document the operator types into the Radar "Policy" tab. It declares which detected agents and tools are allowed, and overrides their risk tier. It is stored in Postgres and applied as an override on top of the system's automatic decisions — it never replaces the gate, quarantine, or token budgets, it only tightens or relaxes within them.

In one line: a human-authored JSON file that decides "allow / deny / what risk" per agent and per tool, applied live by the control plane.

The decision model

Every agent and tool goes through the same three-step model:

┌──────────────────────────────────────────────────────────────────────┐ │ 1. BASELINE The OPA agent assigns a risk tier first. │ │ • tools → a small language model (SLM) reasons the tier │ │ • agents → the authoritative (declared + magazine) tier │ └──────────────────────────────────────────────────────────────────────┘ │ ┌──────────────────────────────────────────────────────────────────────┐ │ 2. YOUR JSON The policy UPDATES that tier: │ │ • "risk": "low" → relabel the tier │ │ • "allow": true/false → hard override (beats everything) │ │ • omit / "default" → no opinion, keep the baseline │ └──────────────────────────────────────────────────────────────────────┘ │ ┌──────────────────────────────────────────────────────────────────────┐ │ 3. SLIDER The "allowed risk level" slider applies it: │ │ tier BELOW the slider → ALLOWED │ │ tier AT OR ABOVE → DENIED │ └──────────────────────────────────────────────────────────────────────┘

So: OPA baseline → your risk → the slider decides, unless an explicit allow short-circuits it. Lower an entry's risk to admit it under the slider; raise it to push it over.

The JSON format

src/awcp/radar/policy.example.json (the template; live copy lives in Postgres) 
{
  "agents": {
    "agent-langgraph-*": { "risk": "low" },          # relabel -> recognised under the slider
    "*temporal*":        { "allow": false }            # explicit hard deny
  },
  "tools": {
    "web_search":  { "risk": "low" },                  # OPA tier -> relabel low -> allowed
    "run_command": { "risk": "severe", "allow": false }# severe tier + hard pre-exec deny
  },
  "skills": {
    "external_post": { "allow": false }                # any agent that CAN post -> deny (see §C)
  }
}
  • Keys match an agent id/name or a tool name; globs like agent-crewai-* are allowed.
  • Tiers: agents low|medium|high; tools also severe.
  • Omitting a field, or the literal string "default", means "use the OPA baseline / slider".
  • Nothing is hardcoded — every agent/tool/skill name comes from this document, never from code.

Storage & enforcement

Storage (Postgres, append-only)

Saved into governance.operator_policy — a versioned, append-only table; the active policy is the most-recent row. The table is created automatically on an existing database (same self-healing pattern as the rest of the schema). An in-process cache keeps the gate hot-path fast; it refreshes on every save.

src/awcp/radar/db.py · operator_policy.py 
# db.py — the durable store
ensure_operator_policy_table()   # CREATE TABLE IF NOT EXISTS governance.operator_policy
save_operator_policy(policy, ...) # INSERT a new version, RETURNING id/ts/version
load_operator_policy()            # the most-recent row

# operator_policy.py — the layer the control plane reads
enabled()           # is a non-empty policy stored?  (inert when not)
agent_recognised()  # allow/deny for an agent (explicit, skill, or slider)
tool_decision()     # relabel + allow/deny for a tool

Where it's enforced

LayerWhat the policy does
Write-action gate (every tool call & agent action)Denies an agent the policy doesn't recognise, and a tool the policy explicitly denies — before execution. Only ever tightens an "allow" into a "deny".
Agent risk (authoritative_risk)An operator risk override wins over the declared/magazine tier (and feeds the slider check + budgets).
OPA agent (tool tiering)Reads the same policy table; after the SLM tiers a tool, it relabels the tier / forces allow-deny so the Tool Risk Tiers view reflects the policy.
src/awcp/radar/api.py — the gate admission (tightens only) 
def _operator_policy_gate(entry, decision, action=""):
    # AGENT: not recognised by the policy -> deny
    if operator_policy.agent_recognised(entry.id, entry.name, risk, threshold, tiers, entry.skills) is False:
        return {**decision, "decision": "deny", "reason": "agent not recognised by operator policy"}
    # TOOL: the gated action IS the tool name -> deny if the policy denies it
    if action and operator_policy.tool_decision(action).get("block") is True:
        return {**decision, "decision": "deny", "reason": f"tool '{action}' denied by operator policy"}
    return decision
Inert by default. With no policy stored, every check returns "no opinion" and nothing changes — detection, onboarding and the gate behave exactly as before. The policy only ever adds constraints you write.

The editor UI (Policy tab)

The React dashboard gained a Policy tab (ui/src/pages/Policy.jsx) — a small code editor for the JSON plus a live "Agent recognition" table. Everything below is one file, no editor library pulled in:

  • Line-number gutter — synced to the editor's scroll.
  • Live JSON syntax colouring — keys, strings, numbers, booleans, punctuation (a transparent textarea over a highlighted layer).
  • Tab / Shift+Tab indent like a real editor.
  • Error line marked red with a wavy underline, and a line N, col M message on Save.
  • Import / Format / Reload buttons; Save validates then writes to Postgres.
  • Agent recognition table — each agent's status, recognised/not, authoritative risk, and (from the card) its Skills.

Before / after & how to check

Before

  • Agents were governed only by the magazine + quarantine + token budgets.
  • No way for an operator to say "allow this agent / deny this tool / set this risk" from one place.
  • A blocked query gave no operator-visible reason tied to a rule.

After

  • One JSON document decides allow/deny + risk per agent and per tool.
  • The gate denies un-recognised agents and denied tools, with a clear reason.
  • The UI shows each agent as recognised / not recognised with its effective risk.

B. AgentCard — what it is

An AgentCard is the A2A-standard manifest an agent publishes at /.well-known/agent.json: its name, version, endpoint, and a typed list of skills. The registry knew what an agent was allowed to do but had no record of what it is for — the card fills that gap, as a pure add-on.

agent serves /.well-known/agent.json │ the radar fetches it via... (a) onboarding (step 1 of 5) (b) scanner (passive) (c) register-card / refresh └──────────────┬───────────────────┬───────────────────┘ ▼ AgentEntry.card / .skills + registry.agents (Postgres) │ GET /agents (+card_summary) · GET /agents/{id}/card · GET /agents?skill=web_search │ React "Radar" page → Skills column + a "card" badge

The shared fetch helper (best-effort, SSRF-guarded)

src/awcp/radar/onboarding.py 
async def fetch_card(entry):
    card_url = entry.endpoint.rstrip("/") + "/.well-known/agent.json"
    assert_safe_url(card_url)                    # refuse private/internal targets
    raw = await httpx.get(card_url)              # 10s timeout
    return raw, skill_ids(raw), note             # on ANY failure -> (None, [], note); never raises

Called from: onboarding (a new fetch_card Temporal activity, run as step 1 of a now-5-step workflow, and the inline fallback); the scanner (sync, once per process); and the two endpoints below. A missing card never blocks onboarding.

The safety boundary & de-duplication

A card is self-published, so its governance fields are advisory only. write_scopes, risk, etc. inside a card are stored for an operator to see, but are never applied to the enforced fields. Proven in testing: a card declaring "write_scopes":["DANGER"] left entry.write_scopes == [] — the card was ignored. An agent cannot widen its own permissions.

De-duplication. The same process used to appear twice — a scanned proc-… row and a self-registered agent-… row. The scanner now folds a scanned process into the self-registered agent that is the same process — matched by pid, then by a shared endpoint / card URL — so one agent = one row. Off-switch: AGENT_RADAR_DEDUP_SCAN=false.

C. The link — match policy rules by skill

The two features connect at one point: the policy's skills section matches any agent by a skill its card declares. Because skills are self-declared, a skill rule is tighten-only — it may deny or raise risk, but can never whitelist or lower (the validator rejects allow:true there). A rule that names an agent directly always wins over a skill rule.

"skills": {
  "external_post": { "allow": false },   # any agent whose card declares this skill -> deny
  "run_command":   { "risk": "high" }    # -> raise its risk tier (can only go up)
}
Proven live (A/B toggle, same agent, same risk, slider raised so only the rule decides): 
skills.external_post rule PRESENT  -> recognised: False
rule REMOVED (only change)         -> recognised: True
rule RESTORED                      -> recognised: False
The card-declared skill is exactly what the policy acts on.

D. How to run it

  1. Start the stack (Postgres, Temporal, Ollama, the OPA agent, the gateway, the UI): 
    cd awcp-mcp-temp
bash scripts/run_everything.sh        # SKIP_INSTALL=1 to skip pip on an existing venv
    The gateway serves the API on :8000; the React dashboard runs on its Vite port (e.g. 5173/5175).
  2. The DB columns + table self-apply on startup (AgentCard columns, the operator-policy table) — no manual SQL.
  3. Set a policy: open the dashboard → Policy tab → paste from src/awcp/radar/policy.example.json → Save. (Or PUT /policy, shown above.)
  4. Set the "allowed risk level" slider on the Tool Risk Tiers view — it's the default allow/deny line.
  5. Watch a card be discovered: point an agent's endpoint at something that serves /.well-known/agent.json, or use register-card. Skills appear in the Policy tab's Skills column and the Radar page badge.
After editing backend code, restart the gateway. It runs without auto-reload, so new code (and the column migration / new onboarding step) loads only on restart. The features are inert/safe until then.
